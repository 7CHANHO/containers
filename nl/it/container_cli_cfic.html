Installazione del plug-in del servizio IBM Bluemix Container (bx ic)

Per eseguire i comandi, il tuo sistema operativo deve soddisfare questi requisiti. Questa attività include le informazioni per l'installazione di tali requisiti. Le versioni precedenti di questi strumenti non sono supportate per l'utilizzo con il plug-in IBM Bluemix Container Service (bx ic).

  • Docker da versione 1.8 a 1.12
  • CLI Cloud Foundry versione 6.14.0 - 6.22.0
    Nota: Gli utenti MacOS Sierra devono installare la CLI Cloud Foundry versione 6.22.0 o successiva.
  • CLI Bluemix
  • Plug-in IBM Bluemix Container Service

Per installare le CLI:

  1. Installa Docker 1.12. Per i sistemi operativi Windows e OS X, Docker viene installato con Docker Toolbox, il quale include Docker Machine, Docker Engine e Kitematic. Per i sistemi operativi Linux, i meccanismi di installazione variano a seconda del tipo di distribuzione Linux che utilizzi. Fai riferimento ai seguenti link per installare o, se è hai già installato Docker, per eseguire l'aggiornamento a Docker 1.12.
    Nota: Se hai già installato Docker sul tuo computer e desideri aggiornarlo alla versione corrente, puoi saltare questo passo e continuare con l'installazione della CLI Cloud Foundry.
    • OS X

      https://github.com/docker/toolbox/releases/download/v1.12/DockerToolbox-1.12.pkg

    • Windows

      https://github.com/docker/toolbox/releases/download/v1.12/DockerToolbox-1.12.exe

    • Linux

      Fai riferimento alla Documentazione per l'installazione di Docker per trovare le istruzioni su come installare la versione di Docker 1.12 sulla distribuzione Linux che utilizzi.

  2. Avvia Docker.
    • Per OS X e Windows, fai doppio clic sull'icona Docker Quickstart Terminal. Il daemon Docker viene avviato e ti vengono configurate le variabili di ambiente.
    • Per Linux, in un programma di utilità della riga di comando, verifica che il daemon Docker sia in esecuzione. Inoltre, configura il daemon Docker per essere riavviato automaticamente. Per ulteriori informazioni, vedi la documentazione Docker per il tuo sistema operativo.
  3. Verifica che la tua installazione Docker sia configurata correttamente eseguendo docker run hello-word nel tuo terminale. 
    docker run hello-world
    Docker cerca l'immagine hello-world localmente da cui eseguire un contenitore, se questa è la prima volta che esegui il comando, riceverai il seguente messaggio. 
    Unable to find image 'hello-world:latest' locally
    L'immagine viene estratta da Docker Hub, un repository pubblico di immagini; al completamento del comando, il tuo output si presenterà così: 
    $ docker run hello-world

Hello from Docker.
This message shows that your installation appears to be working correctly.

To generate this message, Docker took the following steps:
 1. The Docker client contacted the Docker daemon.
 2. The Docker daemon pulled the "hello-world" image from the Docker Hub.
 3. The Docker daemon created a new container from that image which runs the
    executable that produces the output you are currently reading.
 4. The Docker daemon streamed that output to the Docker client, which sent it
    to your terminal.

To try something more ambitious, you can run an Ubuntu container with:
 $ docker run -it ubuntu bash

Share images, automate workflows, and more with a free Docker Hub account:
 https://hub.docker.com

For more examples and ideas, visit:
 https://docs.docker.com/userguide/
    Se il comando non è stato completato correttamente, esegui docker version per controllare se Docker è installato correttamente.
  4. Installa la versione CLI di Cloud Foundry CLI Cloud Foundry versione 6.14.0 - 6.22.0 dal repository GitHub. Devi installare la CLI di Cloud Foundry nella posizione predefinita per il tuo sistema operativo, altrimenti la variabile di ambiente PATH non corrisponde alla tua directory di installazione.
    Nota: Gli utenti MacOS Sierra devono installare la CLI Cloud Foundry versione 6.22.0 o successiva.
    Per verificare che la CLI di Cloud Foundry sia installata correttamente o per verificare la versione qualora avessi già installato Cloud Foundry, immetti il seguente comando:
    cf -v
  5. Installa la CLI CLI Bluemix.
  6. Accedi alla CLI Bluemix. 
    bx login -a https://api.ng.bluemix.net
  7. Installa il plug-in IBM Bluemix Container Service. 
    bx plugin install IBM-Containers -r Bluemix
    Per verificare che il plugin è installato correttamente, esegui il seguente comando:
    bx plugin list

    Il plug-in viene visualizzato nei risultati come IBM - Containers.

Sei pronto ad accedere e a iniziare a gestire i contenitori.

Accesso al plug-in CLI IBM Bluemix Container Service (bx ic)

Dopo che hai installato la CLI, esegui l'accesso per utilizzarla.

Prima di iniziare, completa le seguenti attività.
  • Installa il plug-in IBM Bluemix Container Service (bx ic).
  • Decidi quale organizzazione e spazio Bluemix utilizzare quando accedi a Cloud Foundry. Per ulteriori informazioni sulle organizzazioni e spazi, consulta Organizzazioni e spazi.
  • Per ulteriori informazioni sulle opzioni che puoi specificare con il comando login di Cloud Foundry, vedi il comando di accesso nella documentazione Cloud Foundry.
  1. Apri il programma di utilità della riga di comando
    • Per OS X e Windows, fai doppio clic sull'icona Docker Quickstart Terminal. Il daemon Docker viene avviato e ti vengono configurate le variabili di ambiente.
    • Per Linux, apri qualsiasi programma di utilità della riga di comando.
  2. Accedi a Bluemix tramite la CLI di Cloud Foundry. Per indicare una regione specifica, includere l'opzione -a con l'endpoint API per la regione. Se è la prima volta che utilizzi la CLI e non includi l'opzione, ti viene richiesto di immettere un endpoint API. Una volta specificato l'endpoint API, non devi più immetterlo nei successivi accessi. 
    bx login [-a api.DomainName] [-sso]
    Per accedere a una specifica regione, includi l'endpoint API per quella regione quando esegui l'accesso alla CLI Cloud Foundry.
    • Stati Uniti Sud
      bx login -a api.ng.bluemix.net
    • Regno Unito
      bx login -a api.eu-gb.bluemix.net
    Il parametro SSO (Single Sign On) --sso è obbligatorio quando accedi con un ID federato. Se questa opzione viene utilizzata, apri https://login.ng.bluemix.net/UAALoginServerWAR/passcode per ottenere il passcode monouso. Se non includi l'opzione --sso, completa questi passi secondari.
    1. Per Email, immetti il tuo ID IBM per Bluemix.
    2. Per Password, immetti la password per l'ID IBM. Vengono richiamati i tuoi spazi e organizzazioni Bluemix.
    3. Immetti il numero che rappresenta una delle tue organizzazioni Bluemix.
    4. Immetti il numero che rappresenta uno dei tuoi spazi Bluemix esistenti.
  3. Imposta uno spazio dei nomi per la tua organizzazione. Uno spazio dei nomi è un nome univoco per identificare il tuo registro delle immagini Bluemix privato. Quando crei un contenitore, devi specificare l'ubicazione di un'immagine includendo lo spazio dei nomi con il nome dell'immagine.
    Nota: Quando crei uno spazio dei nomi, non è possibile modificarlo dopo che è stato creato; è solo possibile eliminarlo. Se gli utenti nella tua organizzazione Bluemix hanno già distribuito dei contenitori e impostato uno spazio dei nomi, viene visualizzato un messaggio che mostra lo spazio dei nomi per la tua organizzazione. In tale caso, puoi continuare al passo successivo.
    bx ic namespace-set <mio_spazionomi>
    Suggerimento: Osserva le seguenti regole per lo spazio dei nomi:
    • Può contenere solo lettere minuscole, numeri o caratteri di sottolineatura (_).
    • Può contenere tra 4 e 30 caratteri. Se intendi gestire i contenitori dalla riga di comando, è preferibile utilizzare uno spazio dei nomi breve che possa essere digitato rapidamente.
    • Il suo nome deve essere univoco in Bluemix.
  4. Inizializza IBM Bluemix Container Service. 
    bx ic init
  5. Decidi se desideri eseguire i comandi Docker nativi o bx ic. Puoi selezionare una di queste opzioni ogni volta che accedi al plug-in oppure puoi utilizzare il comando shellinit affinché selezioni sempre l'opzione 2 in automatico.
    • Opzione 1 (Predefinita)

      Una semplice shell per IBM Bluemix Container Service include Docker. IBM Bluemix Container Service non influisce sul daemon Docker locale. I comandi bx ic vengono utilizzati per gestire i contenitori nel tuo spazio Bluemix e i comandi docker vengono utilizzati per gestire i contenitori in locale. I comandi docker sono però sempre richiesti per lo sviluppo di immagini locali.

      Ad esempio, con l'Opzione 1, il comando bx ic run crea un contenitore nel tuo spazio Bluemix e docker run crea un contenitore in locale. Potresti preferire questa opzione se non hai molta familiarità con Docker e IBM Bluemix Container Service o se desideri testare i contenitori in locale.

    • Opzione 2

      Reindirizza il daemon Docker locale per la sessione CLI corrente per IBM Bluemix Container Service in modo che i comandi docker possano essere utilizzati al posto dei comandi bx ic ogni qualvolta siano disponibili per gestire i contenitori nel tuo spazio Bluemix.

      Ad esempio, con l'Opzione 2, è possibile utilizzare il comando bx ic run o docker run per creare un contenitore nel tuo spazio Potresti preferire questa opzione se sei un utente Docker esperto che ha più familiarità con i comandi docker o se stai adattando degli script automatizzati per funzionare con IBM Bluemix Container Service.

      Alcune eccezioni richiedono tuttavia l'utilizzo dei comandi bx ic per l'esecuzione di attività specifiche di IBM Bluemix Container Service. Ad esempio, per impostare uno spazio dei nomi, devi utilizzare il comando bx ic namespace-set. docker namespace-set non è un comando valido. Per un elenco dei comandi docker supportati con questa configurazione, vedi Comandi Docker supportati per il plug-in IBM Bluemix Container Service (bx ic).

      Per configurare questa opzione, copia e incolla le variabili di ambiente fornite nel terminale durante l'accesso.
      • DOCKER_HOST: imposta l'host Docker su IBM Bluemix Container Service.
      • DOCKER_CERT_PATH: imposta l'ubicazione delle chiavi di autenticazione per il plug-in IBM Bluemix Container Service per Cloud Foundry.
      • DOCKER_TLS_VERIFY: utilizza TLS e verifica il server remoto.
      Suggerimento: Quando stai completando le attività nella documentazione IBM Bluemix Container Service, puoi eseguire i comandi docker in tutte le fasi contrassegnate con un asterisco (*).

bx ic attach [--no-stdin] [--sig-proxy] CONTAINER

Scopo
Controllare un contenitore in esecuzione o visualizzarne l'output. Utilizza CTRL+C per chiudere e arrestare il contenitore. Per ulteriori informazioni, vedi il comando attach nella guida di Docker.
Parametri
--no-stdin
(Facoltativo) Non includere l'input standard.
--sig-proxy
(Facoltativo) Il valore predefinito è true.
CONTAINER
(Obbligatorio) Il nome o ID di un contenitore.
Esempio
Il seguente esempio è una richiesta per il collegamento al contenitore mio_contenitore.
bx ic attach mio_contenitore

bx ic build -t TAG [--no-cache] [--pull] [-q] DOCKERFILE_LOCATION

Scopo
Crea un'immagine Docker in locale o nel registro delle immagini Bluemix privato. Per ulteriori informazioni, vedi il comando build nella guida di Docker.
Parametri
DOCKERFILE_LOCATION
(Obbligatorio) Il percorso del Dockerfile e del contesto sull'host locale.
--no-cache
(Facoltativo) Quando un'immagine viene creata per la prima volta, i livelli per l'immagine vengono memorizzati nella cache per un'intera organizzazione. Quando crei le versioni successive dell'immagine, tali versioni memorizzate in cache dell'immagine vengono utilizzate per accelerare il processo di creazione. Includendo l'opzione --no-cache quando crei una nuova versione dell'immagine, puoi garantire che venga ricreato anche ciascun livello dell'immagine.
--pull
(Facoltativo) Per impostazione predefinita, l'immagine di base viene estratta dal registro solo se non è già presente nella cache delle immagini della tua organizzazione. Includendo questa opzione, puoi garantire che per creare la nuova immagine venga utilizzata la versione più recente dell'immagine di base.
-q, --quiet
(Facoltativo) Eliminare l'output dettagliato che viene generato dai contenitori. Il valore predefinito è false.
-t TAG, --tag TAG
(Obbligatorio) Il percorso al tuo registro delle immagini privato da applicare all'immagine creata.
Esempio
Il seguente esempio è una richiesta di creazione di un'immagine denominata myimage. Il Dockerfile e altre risorse utente da utilizzare nella creazione si trovano nella stessa directory da cui viene eseguito il comando. Poiché il registro e lo spazio dei nomi sono inclusi nel nome dell'immagine, l'immagine viene creata nel registro delle immagini Bluemix privato della tua organizzazione.
Suggerimento: Esegui bx ic namespace-get per richiamare il tuo spazio dei nomi e sostituisci <mio_spazionomi> con le informazioni del tuo spazio dei nomi.
bx ic build -t registry.DomainName/<my_namespace>/myimage .

bx ic cp SRC_PATH CONTAINER:DEST_PATH | bx ic cp CONTAINER:SRC_PATH DEST_PATH

Scopo
Copiare un file o una cartella da un contenitore singolo in esecuzione o arrestato sulla macchina locale e viceversa. Per ulteriori informazioni, vedi il comando cp nella guida di Docker.
Nota: Questo comando è supportato solo per essere utilizzato con i singoli contenitori. Quando desideri copiare i file da o in un gruppo di contenitori, devi copiare i file da o in ogni istanza del contenitore del gruppo.
Parametri
DEST_PATH
(Obbligatorio) Il percorso del file o della cartella sulla macchina locale o sul contenitore in cui vuoi copiare le informazioni dal percorso di origine. Il file o la cartella deve esistere sulla macchina locale o sul contenitore prima di poter iniziare la copia. Se vuoi effettuare la copia nella macchina locale, puoi utilizzare il percorso assoluto o relativo. Se vuoi effettuare la copia nel tuo contenitore singolo, devi includere il nome del contenitore, il delimitatore ":" e il percorso del file o della cartella di destinazione. Il percorso deve essere relativo alla directory root del contenitore.
SRC_PATH
(Obbligatorio) Il percorso del file o della cartella che desideri copiare. Se il file o la cartella si trova sulla macchina locale, puoi utilizzare il percorso assoluto o relativo. Se il file o la cartella si trova nel tuo contenitore singolo, devi includere il nome del contenitore, il delimitatore ":" e il percorso del file o cartella che desideri copiare. Il percorso deve essere relativo alla directory root del contenitore.
Nota: Non puoi copiare alcuni file di sistema memorizzati sotto la cartella /dev, /etc, /proc e /sys del tuo contenitore.
Esempio
Questo comando di esempio copia un file denominato config.txt dalla directory corrente della macchina locale alla directory config di un contenitore singolo denominato mio_contenitore.
bx ic cp ./config.txt mio_contenitore:/config

Parametri

SRC
(Obbligatorio) Il nome dell'immagine e repository di origine.

bx ic exec [-d] [-it ] [-u USER] CONTAINER CMD

Scopo
Eseguire un comando all'interno di un contenitore. Per ulteriori informazioni, vedi il comando exec nella guida di Docker.
Parametri
CMD
(Obbligatorio) Il comando da eseguire all'interno del contenitore o dei contenitori specificati.
CONTAINER
(Obbligatorio) Il nome o ID di un contenitore.
-d , --detach
(Facoltativo) Esegui il comando specificato in background.
-it
(Facoltativo) Modalità interattiva. Mantieni la visualizzazione di input standard. Immetti exit per uscire.
-u USER, --user USER
(Facoltativo) Un nome utente.
Esempi
Questo comando di esempio esegue un terminale bash interattivo nel contenitore mio_contenitore.
bx ic exec -it mio_contenitore bash
Questo comando di esempio esegue un comando date nel contenitore mio_contenitore. 
bx ic exec mio_contenitore date

bx ic group-create [--anti] [--auto] [-d DOMAIN] [--desired DESIRED] [-e ENV] [--env-file ENVFILE] [--http_monitor_enabled] [--http_monitor_path] [--http_monitor_rc_list] [--ip IP_ADDRESS] [-m MEMORY] [--max MAX] [--min MIN] [-n HOST] --name NAME [-p PORT] [--session-affinity] [--volume VOLUME:/DIRECTORY_PATH] IMAGE [CMD]

Scopo
Creare un gruppo di scaling. Per impostazione predefinita, i contenitori in Bluemix vengono eseguiti in modalità scollegata.
Parametri
--anti
(Facoltativo) Utilizza l'anti-affinità per rendere il tuo gruppo di contenitori altamente disponibile. L'opzione --anti forza tutte le istanze del contenitore nel tuo gruppo ad essere posizionate in un nodo di calcolo fisico separato, che riduce la probabilità che tutti i contenitori in un gruppo abbiano un arresto anomalo a causa di un malfunzionamento dell'hardware. Puoi non essere in grado utilizzare questa opzione con un gruppo di grandi dimensioni perché ogni regione e organizzazione Bluemix dispone di una serie limitata di nodi di calcolo disponibili per la distribuzione. Se la tua distribuzione non avviene, riduci il numero di istanze del contenitore nel gruppo o rimuovi l'opzione --anti.
--auto
(Facoltativo) Quando il ripristino automatico è abilitato per un gruppo di contenitori e il gruppo è stato in esecuzione per 10 minuti, lo stato del programma di bilanciamento del carico inizia a controllare con regolarità l'integrità di ogni istanza del contenitore nel gruppo tramite le richieste HTTP. Se un'istanza del contenitore non risponde entro 100 secondi, viene contrassegnata come inattiva. Le istanze del contenitore inattive vengono rimosse dal gruppo e ricreate dal ripristino automatico. Il ripristino automatico tenta di ripristinare le istanze del contenitore in un gruppo per 3 volte. Dopo il terzo tentativo, il ripristino automatico non ripristina alcuna istanza del contenitore nel gruppo per 60 minuti. Dopo 60 minuti, il processo di ripristino automatico si riavvia nuovamente.Per ulteriori informazioni sul controllo di integrità, consulta l'opzione --http_monitor_enabled.
CMD
(Facoltativo) Il comando e gli argomenti vengono trasmessi al gruppo di contenitori da eseguire. Questo comando deve essere un comando a esecuzione prolungata. Non utilizzare un comando di breve durata, ad esempio /bin/date, in quanto potrebbe provocare un arresto anomalo del contenitore.
Esempio di comandi a lunga esecuzione che utilizzano l'immagine ibmnode fornita da IBM Bluemix Container Service:
bx ic group-create --name mio_gruppo_di_contenitori registry.DomainName/ibmnode ping localhost
bx ic group-create --name mio_gruppo_di_contenitori registry.DomainName/ibmnode -- tail -f /dev/null
Nota: Il CMD e i suoi argomenti devono trovarsi alla fine della riga di comando bx ic group-create.
-d DOMAIN, --domain DOMAIN
(Facoltativo) Il dominio di sistema predefinito è AppDomainName e fornisce già un certificato SSL; puoi quindi accedere ai tuoi gruppi di contenitori con HTTPS senza alcuna configurazione aggiuntiva. Per utilizzare un dominio personalizzato, devi registrarlo su un server DNS pubblico, configurarlo in Bluemix e associarlo quindi al dominio di sistema Bluemix sul server DNS pubblico. Dopo aver associato il tuo dominio personalizzato al dominio di sistema Bluemix, le richieste per il tuo dominio personalizzato vengono instradate alla tua applicazione in Bluemix. Quando crei un dominio personalizzato, non includere caratteri di sottolineatura (_) nel nome del dominio. Per ulteriori informazioni sui domini personalizzati, vedi Creazione e utilizzo di un dominio personalizzato. Per rendere sicuro il tuo dominio personalizzato, carica un certificato SSL in modo che sia possibile accedere ai gruppi di contenitori con HTTPS.

--desired DESIRED
(Facoltativo) Il numero di istanze nel tuo gruppo. Il valore predefinito è 2. Il numero di istanze del gruppo di contenitori che richiede la tua applicazione dipende dal carico di lavoro previsto e dal tempo nel quale una richiesta può essere elaborata dall'applicazione. Ad esempio, il multitasking o la connettività di rete influisce su quante istanze possono essere richieste per la tua applicazione. Per determinare il numero di istanze desiderato, distribuisci un singolo contenitore in Bluemix che esegue la tua applicazione ed esegue un un test di carico su questo contenitore. Se l'applicazione può gestire, ad esempio, 100 richiesta al secondo, ma prevedi un carico di lavoro di 300 richieste al secondo, il numero desiderato di istanze per il gruppo di contenitori è 3. Per rendere il tuo gruppo di contenitori altamente disponibile e con maggiore resilenza a un malfunzionamento, considera di includere più istanze rispetto al numero minimo per gestire il carico di lavoro previsto. Più istanze possono gestire il carico di lavoro nel caso di un altro arresto anomalo di un'istanza. Per garantire il ripristino automatico di un'istanza con un malfunzionamento senza influire sul carico di lavoro, includi un'ulteriore istanza. Per la protezione da due malfunzionamenti simultanei, includi due ulteriori istanze. Questa configurazione è un modello N+2 dove N è il numero di istanze per gestire le richieste e +2 sono le due istanze aggiuntive.
Importante: Un gruppo di contenitori deve includere almeno 2 istanze del contenitore. Se includi solo 1 istanza nel gruppo e tale istanza diventa non disponibile, i tuoi utenti possono riscontrare dei tempi di inattività per l'applicazione.

--http_monitor_enabled
(Facoltativo) Il valore predefinito è true. Quando il ripristino automatico è abilitato per un gruppo di contenitori e il gruppo è stato in esecuzione per 10 minuti, lo stato del programma di bilanciamento del carico inizia a controllare con regolarità l'integrità di ogni istanza del contenitore nel gruppo tramite le richieste HTTP. Se un'istanza del contenitore non risponde entro 100 secondi, viene contrassegnata come inattiva. Le istanze del contenitore inattive vengono rimosse dal gruppo e ricreate dal ripristino automatico. Il ripristino automatico tenta di ripristinare le istanze del contenitore in un gruppo per 3 volte. Dopo il terzo tentativo, il ripristino automatico non ripristina alcuna istanza del contenitore nel gruppo per 60 minuti. Dopo 60 minuti, il processo di ripristino automatico si riavvia nuovamente. Alcune applicazioni possono non richiedere un controllo di integrità HTTP, come ad esempio un'applicazione che utilizza TCP per la comunicazione invece di HTTP. Per disabilitare il monitoraggio HTTP, imposta l'opzione su false. Esempio: --http_monitor_enabled=false
--http_monitor_path
(Facoltativo) Il valore predefinito è /. Quando --http_monitor_enabled è abilitata, puoi utilizzare questa opzione per impostare un percorso personalizzato per il controllo di integrità, come --http_monitor_path="/health/path".
--http_monitor_rc_list
(Facoltativo) Quando --http_monitor_enabled è abilitata, puoi specificare l'elenco dei codici di risposta per il controllo di integrità. Il valore predefinito include i codici 200, 201, 202, 204, 300, 301, 302, 401, 403 e 404. Tutte le risposte non presenti nell'elenco delle risposte valide modificano lo stato dell'istanza del contenitore in INACTIVE tramite il programma di bilanciamento del carico. Un contenitore non attivo non riceve alcun carico e lo stato del gruppo di contenitori viene modificato in HAS_INACTIVE_INSTANCES. Se nessuno dei contenitori è ACTIVE, lo stato del gruppo di contenitori viene modificato in NO_ACTIVE_INSTANCES. Per ulteriori informazioni sugli stati, consulta Stati del contenitore in IBM Bluemix Container Service.

(Facoltativo) Immetti un limite di memoria per il tuo contenitore in MB. Dopo che una dimensione del contenitore è stata assegnata, non è possibile modificare il valore. Le dimensioni disponibili in IBM Bluemix Container Service sono:
  • Pico (64 MB di memoria, 4 GB di spazio su disco)
  • Nano (128 MB di memoria, 8 GB di spazio su disco)
  • Micro (256 MB di memoria, 16 GB di spazio su disco)
  • Tiny (512 MB di memoria, 32 GB di spazio su disco)
  • Small (1024 MB di memoria, 64 GB di spazio su disco)
  • Medium (2048 MB di memoria, 128 GB di spazio su disco)
  • Large (4096 MB di memoria, 256 GB di spazio su disco)
  • X-Large (8192 MB di memoria, 512 GB di spazio su disco)
  • 2X-Large (16384 MB di memoria, 1 TB di spazio su disco)

Se non hai più bisogno del plug-in IBM Bluemix Container Service, puoi disinstallarlo con un comando bx.

Nota: Per disinstallare il plug-in Cloud Foundry di IBM Bluemix Container Service, esegui questo comando. 
cf uninstall-plugin IBM-Containers
  1. Per disinstallare il plug-in IBM Bluemix Container Service per la CLI Bluemix, esegui questo comando. 
    bx plugin uninstall IBM-Containers
  2. Verifica che il plug-in sia stato disinstallato eseguendo questo comando e controllando l'elenco dei plug-in installati. 
    bx plugin list
