bx ic attach [--no-stdin] [--sig-proxy] CONTAINER

Scopo
Controllare un contenitore in esecuzione o visualizzarne l'output. Utilizza CTRL+C per chiudere e arrestare il contenitore. Per ulteriori informazioni, vedi il comando attach nella guida di Docker.
Parametri
--no-stdin
(Facoltativo) Non includere l'input standard.
--sig-proxy
(Facoltativo) Il valore predefinito è true.
CONTAINER
(Obbligatorio) Il nome o ID di un contenitore.
Esempio
Il seguente esempio è una richiesta per il collegamento al contenitore mio_contenitore.
bx ic attach mio_contenitore

bx ic build -t TAG [--no-cache] [--pull] [-q] DOCKERFILE_LOCATION

Scopo
Crea un'immagine Docker in locale o nel registro delle immagini Bluemix privato. Per ulteriori informazioni, vedi il comando build nella guida di Docker.
Parametri
DOCKERFILE_LOCATION
(Obbligatorio) Il percorso del Dockerfile e del contesto sull'host locale.
--no-cache
(Facoltativo) Quando un'immagine viene creata per la prima volta, i livelli per l'immagine vengono memorizzati nella cache per un'intera organizzazione. Quando crei le versioni successive dell'immagine, tali versioni memorizzate in cache dell'immagine vengono utilizzate per accelerare il processo di creazione. Includendo l'opzione --no-cache quando crei una nuova versione dell'immagine, puoi garantire che venga ricreato anche ciascun livello dell'immagine.
--pull
(Facoltativo) Per impostazione predefinita, l'immagine di base viene estratta dal registro solo se non è già presente nella cache delle immagini della tua organizzazione. Includendo questa opzione, puoi garantire che per creare la nuova immagine venga utilizzata la versione più recente dell'immagine di base.
-q, --quiet
(Facoltativo) Eliminare l'output dettagliato che viene generato dai contenitori. Il valore predefinito è false.
-t TAG, --tag TAG
(Obbligatorio) Il percorso al tuo registro delle immagini privato da applicare all'immagine creata.
Esempio
Il seguente esempio è una richiesta di creazione di un'immagine denominata myimage. Il Dockerfile e altre risorse utente da utilizzare nella creazione si trovano nella stessa directory da cui viene eseguito il comando. Poiché il registro e lo spazio dei nomi sono inclusi nel nome dell'immagine, l'immagine viene creata nel registro delle immagini Bluemix privato della tua organizzazione.
Suggerimento: Esegui bx ic namespace-get per richiamare il tuo spazio dei nomi e sostituisci <mio_spazionomi> con le informazioni del tuo spazio dei nomi.
bx ic build -t registry.DomainName/<my_namespace>/myimage .

bx ic cp SRC_PATH CONTAINER:DEST_PATH | bx ic cp CONTAINER:SRC_PATH DEST_PATH

Scopo
Copiare un file o una cartella da un contenitore singolo in esecuzione o arrestato sulla macchina locale e viceversa. Per ulteriori informazioni, vedi il comando cp nella guida di Docker.
Nota: Questo comando è supportato solo per essere utilizzato con i singoli contenitori. Quando desideri copiare i file da o in un gruppo di contenitori, devi copiare i file da o in ogni istanza del contenitore del gruppo.
Parametri
DEST_PATH
(Obbligatorio) Il percorso del file o della cartella sulla macchina locale o sul contenitore in cui vuoi copiare le informazioni dal percorso di origine. Il file o la cartella deve esistere sulla macchina locale o sul contenitore prima di poter iniziare la copia. Se vuoi effettuare la copia nella macchina locale, puoi utilizzare il percorso assoluto o relativo. Se vuoi effettuare la copia nel tuo contenitore singolo, devi includere il nome del contenitore, il delimitatore ":" e il percorso del file o della cartella di destinazione. Il percorso deve essere relativo alla directory root del contenitore.
SRC_PATH
(Obbligatorio) Il percorso del file o della cartella che desideri copiare. Se il file o la cartella si trova sulla macchina locale, puoi utilizzare il percorso assoluto o relativo. Se il file o la cartella si trova nel tuo contenitore singolo, devi includere il nome del contenitore, il delimitatore ":" e il percorso del file o cartella che desideri copiare. Il percorso deve essere relativo alla directory root del contenitore.
Nota: Non puoi copiare alcuni file di sistema memorizzati sotto la cartella /dev, /etc, /proc e /sys del tuo contenitore.
Esempio
Questo comando di esempio copia un file denominato config.txt dalla directory corrente della macchina locale alla directory config di un contenitore singolo denominato mio_contenitore.
bx ic cp ./config.txt mio_contenitore:/config

bx ic cpi SRC DST

Scopo
Accedere a un'immagine Docker Hub o a un'immagine del tuo registro locale e copiarla nel tuo registro delle immagini Bluemix privato.
Nota: Questo comando è un'alternativa all'utilizzo congiunto di docker pull e docker push per aggiungere un'immagine al tuo registro delle immagini Bluemix. Se preferisci utilizzare docker pull e docker push, vedi Esecuzione del push delle immagini locali al tuo registro Bluemix privato dalla riga di comando.
Parametri
DST
(Obbligatorio) L'URL del registro delle immagini Bluemix privato, che include lo spazio dei nomi e il nome dell'immagine di destinazione. La tag per l'immagine è facoltativa.
SRC
(Obbligatorio) Il nome dell'immagine e repository di origine.
Esempio
bx ic cpi repository_origine/nome_immagine_origine URL_registro_privato/nome_immagine_di_destinazione:tag
Suggerimento: Esegui bx ic namespace-get per richiamare il tuo spazio dei nomi e sostituisci <mio_spazionomi> con le informazioni del tuo spazio dei nomi.
bx ic cpi training/sinatra registry.DomainName/<mio_spazionomi>/mysinatra:v1

bx ic exec [-d] [-it ] [-u USER] CONTAINER CMD

Scopo
Eseguire un comando all'interno di un contenitore. Per ulteriori informazioni, vedi il comando exec nella guida di Docker.
Parametri
CMD
(Obbligatorio) Il comando da eseguire all'interno del contenitore o dei contenitori specificati.
CONTAINER
(Obbligatorio) Il nome o ID di un contenitore.
-d , --detach
(Facoltativo) Esegui il comando specificato in background.
-it
(Facoltativo) Modalità interattiva. Mantieni la visualizzazione di input standard. Immetti exit per uscire.
-u USER, --user USER
(Facoltativo) Un nome utente.
Esempi
Questo comando di esempio esegue un terminale bash interattivo nel contenitore mio_contenitore.
bx ic exec -it mio_contenitore bash
Questo comando di esempio esegue un comando date nel contenitore mio_contenitore. 
bx ic exec mio_contenitore date

bx ic group-create [--anti] [--auto] [-d DOMAIN] [--desired DESIRED] [-e ENV] [--env-file ENVFILE] [--http_monitor_enabled] [--http_monitor_path] [--http_monitor_rc_list] [--ip IP_ADDRESS] [-m MEMORY] [--max MAX] [--min MIN] [-n HOST] --name NAME [-p PORT] [--session-affinity] [--volume VOLUME:/DIRECTORY_PATH] IMAGE [CMD]

Scopo
Creare un gruppo di scaling. Per impostazione predefinita, i contenitori in Bluemix vengono eseguiti in modalità scollegata.
Parametri
--anti
(Facoltativo) Utilizza l'anti-affinità per rendere il tuo gruppo di contenitori altamente disponibile. L'opzione --anti forza tutte le istanze del contenitore nel tuo gruppo ad essere posizionate in un nodo di calcolo fisico separato, che riduce la probabilità che tutti i contenitori in un gruppo abbiano un arresto anomalo a causa di un malfunzionamento dell'hardware. Puoi non essere in grado utilizzare questa opzione con un gruppo di grandi dimensioni perché ogni regione e organizzazione Bluemix dispone di una serie limitata di nodi di calcolo disponibili per la distribuzione. Se la tua distribuzione non avviene, riduci il numero di istanze del contenitore nel gruppo o rimuovi l'opzione --anti.
--auto
(Facoltativo) Quando il ripristino automatico è abilitato per un gruppo di contenitori e il gruppo è stato in esecuzione per 10 minuti, lo stato del programma di bilanciamento del carico inizia a controllare con regolarità l'integrità di ogni istanza del contenitore nel gruppo tramite le richieste HTTP. Se un'istanza del contenitore non risponde entro 100 secondi, viene contrassegnata come inattiva. Le istanze del contenitore inattive vengono rimosse dal gruppo e ricreate dal ripristino automatico. Il ripristino automatico tenta di ripristinare le istanze del contenitore in un gruppo per 3 volte. Dopo il terzo tentativo, il ripristino automatico non ripristina alcuna istanza del contenitore nel gruppo per 60 minuti. Dopo 60 minuti, il processo di ripristino automatico si riavvia nuovamente.Per ulteriori informazioni sul controllo di integrità, consulta l'opzione --http_monitor_enabled.
CMD
(Facoltativo) Il comando e gli argomenti vengono trasmessi al gruppo di contenitori da eseguire. Questo comando deve essere un comando a esecuzione prolungata. Non utilizzare un comando di breve durata, ad esempio /bin/date, in quanto potrebbe provocare un arresto anomalo del contenitore.
Esempio di comandi a lunga esecuzione che utilizzano l'immagine ibmnode fornita da IBM Bluemix Container Service:
bx ic group-create --name mio_gruppo_di_contenitori registry.DomainName/ibmnode ping localhost
bx ic group-create --name mio_gruppo_di_contenitori registry.DomainName/ibmnode -- tail -f /dev/null
Nota: Il CMD e i suoi argomenti devono trovarsi alla fine della riga di comando bx ic group-create.
-d DOMAIN, --domain DOMAIN
(Facoltativo) Il dominio di sistema predefinito è AppDomainName e fornisce già un certificato SSL; puoi quindi accedere ai tuoi gruppi di contenitori con HTTPS senza alcuna configurazione aggiuntiva. Per utilizzare un dominio personalizzato, devi registrarlo su un server DNS pubblico, configurarlo in Bluemix e associarlo quindi al dominio di sistema Bluemix sul server DNS pubblico. Dopo aver associato il tuo dominio personalizzato al dominio di sistema Bluemix, le richieste per il tuo dominio personalizzato vengono instradate alla tua applicazione in Bluemix. Quando crei un dominio personalizzato, non includere caratteri di sottolineatura (_) nel nome del dominio. Per ulteriori informazioni sui domini personalizzati, vedi Creazione e utilizzo di un dominio personalizzato. Per rendere sicuro il tuo dominio personalizzato, carica un certificato SSL in modo che sia possibile accedere ai gruppi di contenitori con HTTPS.

L'host e il dominio, combinati, formano l'URL di rotta pubblico completo, come http://mycontainerhost.AppDomainName, e devono essere univoci in Bluemix. Quando esamini i dettagli di un gruppo di contenitori con il comando bx ic group-inspect, l'host e il dominio sono elencati insieme come rotta.

--desired DESIRED
(Facoltativo) Il numero di istanze nel tuo gruppo. Il valore predefinito è 2. Il numero di istanze del gruppo di contenitori che richiede la tua applicazione dipende dal carico di lavoro previsto e dal tempo nel quale una richiesta può essere elaborata dall'applicazione. Ad esempio, il multitasking o la connettività di rete influisce su quante istanze possono essere richieste per la tua applicazione. Per determinare il numero di istanze desiderato, distribuisci un singolo contenitore in Bluemix che esegue la tua applicazione ed esegue un un test di carico su questo contenitore. Se l'applicazione può gestire, ad esempio, 100 richiesta al secondo, ma prevedi un carico di lavoro di 300 richieste al secondo, il numero desiderato di istanze per il gruppo di contenitori è 3. Per rendere il tuo gruppo di contenitori altamente disponibile e con maggiore resilenza a un malfunzionamento, considera di includere più istanze rispetto al numero minimo per gestire il carico di lavoro previsto. Più istanze possono gestire il carico di lavoro nel caso di un altro arresto anomalo di un'istanza. Per garantire il ripristino automatico di un'istanza con un malfunzionamento senza influire sul carico di lavoro, includi un'ulteriore istanza. Per la protezione da due malfunzionamenti simultanei, includi due ulteriori istanze. Questa configurazione è un modello N+2 dove N è il numero di istanze per gestire le richieste e +2 sono le due istanze aggiuntive.
Importante: Un gruppo di contenitori deve includere almeno 2 istanze del contenitore. Se includi solo 1 istanza nel gruppo e tale istanza diventa non disponibile, i tuoi utenti possono riscontrare dei tempi di inattività per l'applicazione.
-e ENV, --env ENV
(Facoltativo) Imposta la variabile di ambiente dove ENV è una coppia key=value. Elenca più chiavi separatamente e, se includi delle virgolette, includile a racchiudere sia il nome che il valore delle variabili di ambiente. Esempio: -e "chiave1=valore1" -e "chiave2=valore2a,valore2b" -e "chiave3=valore3"

--http_monitor_enabled
(Facoltativo) Il valore predefinito è true. Quando il ripristino automatico è abilitato per un gruppo di contenitori e il gruppo è stato in esecuzione per 10 minuti, lo stato del programma di bilanciamento del carico inizia a controllare con regolarità l'integrità di ogni istanza del contenitore nel gruppo tramite le richieste HTTP. Se un'istanza del contenitore non risponde entro 100 secondi, viene contrassegnata come inattiva. Le istanze del contenitore inattive vengono rimosse dal gruppo e ricreate dal ripristino automatico. Il ripristino automatico tenta di ripristinare le istanze del contenitore in un gruppo per 3 volte. Dopo il terzo tentativo, il ripristino automatico non ripristina alcuna istanza del contenitore nel gruppo per 60 minuti. Dopo 60 minuti, il processo di ripristino automatico si riavvia nuovamente. Alcune applicazioni possono non richiedere un controllo di integrità HTTP, come ad esempio un'applicazione che utilizza TCP per la comunicazione invece di HTTP. Per disabilitare il monitoraggio HTTP, imposta l'opzione su false. Esempio: --http_monitor_enabled=false
--http_monitor_path
(Facoltativo) Il valore predefinito è /. Quando --http_monitor_enabled è abilitata, puoi utilizzare questa opzione per impostare un percorso personalizzato per il controllo di integrità, come --http_monitor_path="/health/path".
--http_monitor_rc_list
(Facoltativo) Quando --http_monitor_enabled è abilitata, puoi specificare l'elenco dei codici di risposta per il controllo di integrità. Il valore predefinito include i codici 200, 201, 202, 204, 300, 301, 302, 401, 403 e 404. Tutte le risposte non presenti nell'elenco delle risposte valide modificano lo stato dell'istanza del contenitore in INACTIVE tramite il programma di bilanciamento del carico. Un contenitore non attivo non riceve alcun carico e lo stato del gruppo di contenitori viene modificato in HAS_INACTIVE_INSTANCES. Se nessuno dei contenitori è ACTIVE, lo stato del gruppo di contenitori viene modificato in NO_ACTIVE_INSTANCES. Per ulteriori informazioni sugli stati, consulta Stati del contenitore in IBM Bluemix Container Service.
IMAGE
(Obbligatorio) L'immagine da includere in ogni istanza del contenitore del gruppo di contenitori. Dopo l'immagine puoi elencare comandi, ma non inserire opzioni; includi tutte le opzioni prima di specificare un'immagine.

Se stai utilizzando un'immagine nel registro delle immagini Bluemix privato della tua organizzazione, specifica l'immagine nel seguente formato: registry.DomainName/NAMESPACE/IMAGE

Se stai utilizzando un'immagine fornita da IBM Bluemix Container Service, non includere lo spazio dei nomi della tua organizzazione. Specifica l'immagine nel seguente formato: registry.DomainName/IMAGE

--ip IP_ADDRESS
(Facoltativo) se disponi di un indirizzo IP disponibile, esegui il bind di un indirizzo IP pubblico al tuo gruppo di contenitori. Devi anche esporre la porta con l'indicatore -p. Un gruppo di contenitori può avere anche una rotta pubblica o un indirizzo IP pubblico, ma non entrambi. Per annullare il bind dell'indirizzo IP pubblico e restituire l'IP alla quota dello spazio, rimuovi il gruppo di contenitori e ricrea il gruppo senza l'indirizzo IP. Per richiedere un indirizzo IP prima della creazione di un gruppo di contenitori, esegui bx ic ip-request.
Nota: Il bind a un indirizzo IP pubblico non include un certificato SSL che codifica i dati inviati al e dal tuo gruppo di contenitori. Se la tua applicazione richiede la codifica SSL, puoi anche implementare la tua propria soluzione SSL o associare una rotta pubblica al tuo gruppo di contenitori invece di eseguire il bind a un indirizzo IP pubblico. Le rotte pubbliche già includono un certificato SSL, così puoi accedere al tuo gruppo di contenitori con HTTPS senza ulteriori configurazioni.
-m MEMORY, --memory MEMORY
(Facoltativo) Immetti un limite di memoria per il tuo contenitore in MB. Il limite di memoria è la parte della dimensione del contenitore che definisce la quantità massima di memoria e spazio su disco di un contenitore nell'host di calcolo durante il runtime. Dopo che una dimensione del contenitore è stata assegnata, non è possibile modificare il valore. Le dimensioni disponibili in IBM Bluemix Container Service sono:
  • Pico (64 MB di memoria, 4 GB di spazio su disco)
  • Nano (128 MB di memoria, 8 GB di spazio su disco)
  • Micro (256 MB di memoria, 16 GB di spazio su disco)
  • Tiny (512 MB di memoria, 32 GB di spazio su disco)
  • Small (1024 MB di memoria, 64 GB di spazio su disco)
  • Medium (2048 MB di memoria, 128 GB di spazio su disco)
  • Large (4096 MB di memoria, 256 GB di spazio su disco)
  • X-Large (8192 MB di memoria, 512 GB di spazio su disco)
  • 2X-Large (16384 MB di memoria, 1 TB di spazio su disco)
Se non configuri una dimensione per il tuo contenitore, tutte le istanze del contenitore vengono create con 256 MB.
Importante: Immetti la memoria del contenitore in MB senza l'etichetta dell'unità. Ad esempio, se desideri creare un contenitore pico, immetti -m 64.
--max MAX
(Facoltativo) Il numero massimo di istanze. Il valore predefinito è 2.
--min MIN
(Facoltativo) Il numero minimo di istanze. Il valore predefinito è 1.
-n HOST, --hostname HOST
(Facoltativo) Il nome host, ad esempio mycontainerhost. Non includere i caratteri di sottolineatura (_) nel nome host. L'host e il dominio, combinati, formano l'URL di rotta pubblico completo, come ad esempio http://mycontainerhost.AppDomainName. Quando esamini i dettagli di un gruppo di contenitori con il comando bx ic group-inspect, l'host e il dominio sono elencati insieme come rotta.
--name NAME
(Obbligatorio) Assegnare un nome al gruppo. -n è stato dichiarato obsoleto.
Suggerimento: Il nome contenitore deve iniziare con una lettera e puoi quindi includere lettere maiuscole e minuscole, numeri, punti (.), caratteri di sottolineatura (_) o trattini (-).
-p PORT, --publish PORT

(Obbligatorio) Esporre la porta per il traffico di rete privata e HTTP. Se non esponi una porta, il gruppo di contenitori viene eseguito, ma è impossibile accedervi tramite il programma di bilanciamento del carico di lavoro nella rete privata o pubblica.

Per i gruppi di contenitori, non è possibile includere più porte. Se sono elencate più porte, viene esposta solo l'ultima porta. Le altre porte vengono ignorate dal comando.

Quando esegui il bind a un servizio, i contenitori nel tuo gruppo devono essere in ascolto per il traffico HTTP nella porta presentata del gruppo. Le porte non HTTP non possono essere esposte pubblicamente. Quando il traffico HTTPS arriva alla porta esposta, il (Go)Router completa la terminazione HTTPS. Quindi, viene utilizzato il protocollo HTTP sulla rete virtuale privata tra il (Go)Router e i contenitori. Quando esegui il bind a un indirizzo IP, i contenitori nel tuo gruppo possono essere in ascolto per il traffico HTTP o HTTPs ma possono ancora esporre solo una porta.
Se è specificata una porta nel Dockerfile per l'immagine che stai utilizzando, includi tale porta.
Suggerimento:
  • Per l'immagine Liberty Server certificata da IBM o una versione modificata di questa immagine, immetti la porta 9080.
  • Per l'immagine Node.js certificata da IBM o una versione modificata di questa immagine, immetti una porta qualsiasi, quale ad esempio la porta 8000.
--session_affinity

(Facoltativo) Il valore predefinito è false. Quando --http_monitor_enabled è impostata su true, puoi impostare l'opzione --session_affinity su true per consentire a IBM Bluemix Container Service di creare e gestire un cookie della sessione al posto dell'applicazione. Questa opzione non è disponibile se --http_monitor_enabled è impostata su false o se stai utilizzando TCP per la comunicazione invece di HTTP.

(Facoltativo) Collega un volume a un contenitore specificando i dettagli nel seguente formato VOLUME:/DIRECTORY_PATH[:ro].
  • VOLUME: l'ID o nome del volume.
  • DIRECTORY_PATH: il percorso assoluto alla directory nel contenitore.
  • ro: facoltativo. Se si specifica ro, il volume sarà di sola lettura anziché di lettura/scrittura come da impostazione predefinita.

Nota: Devi assicurarti che lo strumento di comando Cloud Foundry sia installato e di disporre di un token Cloud Foundry. L'esecuzione corretta dell'accesso utilizzando bx login e bx ic init genera il token richiesto.
Parametri
CMD
(Facoltativo) Il comando e gli argomenti sono passati al contenitore per l'esecuzione. Questo comando deve essere un comando a esecuzione prolungata. Non utilizzare un comando di breve durata, ad esempio /bin/date, in quanto potrebbe provocare un arresto anomalo del contenitore.
Esempio di comandi a lunga esecuzione, presumendo che tu stia utilizzando l'immagine ibmnode fornita nel registro IBM Bluemix Container Service:
bx ic run --name mio_contenitore registry.DomainName/ibmnode ping localhost
bx ic run --name mio_contenitore registry.DomainName/ibmnode -- tail -f /dev/null
bx ic run --name mio_contenitore registry.DomainName/ibmnode -- sh -c "while true; do date; sleep 20; done"
Nota: Il CMD e i suoi argomenti devono trovarsi alla fine della riga di comando bx ic group-create .
-d
(Facoltativo) Esegui il contenitore in modalità scollegata. I contenitori vengono eseguiti in modalità scollegata per impostazione predefinita quando utilizzi bx ic run. In modalità scollegata, quando il processo root nel contenitore esegue l'uscita, lo fa anche il contenitore. Ad esempio, quando distribuisci un contenitore dall'immagine ibmnode, il tuo contenitore viene arrestato immediatamente dopo la sua creazione, a condizione che non ci sia alcun processo a lunga esecuzione.
Suggerimento: Se scegli di utilizzare docker run invece di bx ic run, è necessaria l'opzione -d.
-e ENV, --env ENV
(Facoltativo) Imposta il numero di variabili di ambiente dove ENV è una coppia key=value. Elenca più chiavi separatamente e, se includi delle virgolette, includile a racchiudere sia il nome che il valore delle variabili di ambiente.

Esempio

-e "chiave1=valore1" -e "chiave2=valore2" -e "chiave3=valore3"

(Obbligatorio) L'immagine da includere nel contenitore. Dopo l'immagine puoi elencare comandi, ma non inserire opzioni; includi tutte le opzioni prima di specificare un'immagine.

-it
(Facoltativo) Esegui il contenitore in modalità interattiva. Una volta creato il contenitore, l'input standard rimane visualizzato. Immetti exit per uscire.
--link NAME:ALIAS
(Facoltativo) Quando desideri che un contenitore comunichi con un altro contenitore in esecuzione, puoi indirizzarti a esso utilizzando un alias per il nome host. Per ulteriori informazioni, vedi Collegamento dei contenitori singoli con l'interfaccia riga di comando (CLI).
-m MEMORY, --memory MEMORY
(Facoltativo) Immetti un limite di memoria per il tuo contenitore in MB. Il limite di memoria è la parte della dimensione del contenitore che definisce la quantità massima di memoria e spazio su disco di un contenitore nell'host di calcolo durante il runtime. Dopo che una dimensione del contenitore è stata assegnata, non è possibile modificare il valore. Le dimensioni disponibili in IBM Bluemix Container Service sono:
  • Pico (64 MB di memoria, 4 GB di spazio su disco)
  • Nano (128 MB di memoria, 8 GB di spazio su disco)
  • Micro (256 MB di memoria, 16 GB di spazio su disco)
  • Tiny (512 MB di memoria, 32 GB di spazio su disco)
  • Small (1024 MB di memoria, 64 GB di spazio su disco)
  • Medium (2048 MB di memoria, 128 GB di spazio su disco)
  • Large (4096 MB di memoria, 256 GB di spazio su disco)
  • X-Large (8192 MB di memoria, 512 GB di spazio su disco)
  • 2X-Large (16384 MB di memoria, 1 TB di spazio su disco)
Se non configuri una dimensione per il tuo contenitore, tutte le istanze del contenitore vengono create con 256 MB.
Importante: Immetti la memoria del contenitore in MB senza l'etichetta dell'unità. Ad esempio, se desideri creare un contenitore pico, immetti -m 64.
--name NAME
(Obbligatorio) Assegnare un nome al contenitore.
Suggerimento: Il nome contenitore deve iniziare con una lettera e puoi quindi includere lettere maiuscole e minuscole, numeri, punti (.), caratteri di sottolineatura (_) o trattini (-).
-p PORT, --publish PORT

(Facoltativo) Se vuoi rendere la tua applicazione accessibile da Internet, devi esporre una porta pubblica. Quando esponi una porta pubblica, crei un gruppo di sicurezza di rete pubblico per il tuo contenitore che ti consente di inviare e ricevere dati pubblici solo nella porta esposta. Tutte le altre porte pubbliche sono chiuse e non possono essere utilizzate per accedere alla tua applicazione da internet. Puoi includere più porte con più opzioni -p. Le porte non possono essere associate o inoltrate. Se non esponi una porta, il tuo contenitore è accessibile solo dalla rete del contenitore privata. Puoi utilizzare l'indirizzo IP assegnato per comunicare con il tuo contenitore nella rete privata.

Se è specificata una porta nel Dockerfile per l'immagine che stai utilizzando, includi tale porta.
Suggerimento:
  • Per l'immagine Liberty Server certificata da IBM o una versione modificata di questa immagine, immetti la porta 9080.
  • Per l'immagine Node.js certificata da IBM o una versione modificata di questa immagine, immetti una porta qualsiasi, quale ad esempio la porta 8000.

Puoi scegliere tra UDP e TCP per indicare il protocollo IP che vuoi utilizzare. Se non specifichi un protocollo, la tua porta viene automaticamente esposta per il traffico TCP.

Esempi
  • -p 9080
  • -p 9080/udp
  • -p 9080/udp -o 8000/tcp

Se hai un indirizzo IP esistente nello spazio di cui desideri eseguire il bind al contenitore, puoi specificarlo con le informazioni sulla porta. Includi l'indirizzo IP, la porta o l'indirizzo IP e il numero di porta del contenitore, come ad esempio <indirizzo-ip>:<porta_su_IP>:<porta-contenitore>. La porta sull'indirizzo IP deve corrispondere alla porta del contenitore esposta nel Dockerfile.

Esempio
-p 192.0.2.23:9080:9080
Nota: Il bind a un indirizzo IP pubblico non include un certificato SSL che codifica i dati inviati al e dal tuo contenitore singolo. Se la tua applicazione richiede la codifica SSL, puoi anche implementare la tua propria soluzione SSL o utilizzare un gruppo di contenitori invece di un contenitore singolo. I gruppi di contenitori sono associati a una rotta pubblica che già include un certificato SSL, per cui puoi accedere al tuo gruppo di contenitori con HTTPS senza alcuna ulteriore configurazione. Per ulteriori informazioni, consulta Creazione di un gruppo di contenitori con la GUI Bluemix.
-P
(Facoltativo) Esporre automaticamente le porte specificate nel Dockerfile dell'immagine per il traffico HTTP.
--volume VOLUME:/DIRECTORY_PATH[:ro]
(Facoltativo) Collega un volume a un contenitore specificando i dettagli nel seguente formato VOLUME:/DIRECTORY_PATH[:ro].
  • VOLUME Il nome o l'ID del volume.
  • DIRECTORY_PATH: il percorso assoluto alle directory nel contenitore.
  • ro Facoltativo. Se si specifica ro, il volume sarà di sola lettura anziché di lettura/scrittura come da impostazione predefinita.
Esempi
Il seguente esempio è una richiesta valida per creare e avviare un nuovo contenitore utilizzando l'immagine mio_spazionomi/nginx, con il nome mio_contenitore e un limite di memoria di 1024 MB, dove mio_spazionomi è lo spazio dei nomi associato agli utenti collegati.
bx ic run -m 1024 --name mio_contenitore registry.DomainName/mio_spazionomi/nginx

cf ic group create [--anti] [--auto] [-d DOMAIN] [--desired DESIRED] [-e ENV] [--env-file ENVFILE] [--http_monitor_enabled] [--http_monitor_path] [--http_monitor_rc_list] [--ip IP_ADDRESS] [-m MEMORY] [--max MAX] [--min MIN] [-n HOST] --name NAME [-p PORT] [--session_affinity] [--volume VOLUME:/DIRECTORY_PATH] IMAGE [CMD]

Scopo
Creare un gruppo di scaling. Per impostazione predefinita, i contenitori in Bluemix vengono eseguiti in modalità scollegata.
Parametri
--anti
(Facoltativo) Utilizza l'anti-affinità per rendere il tuo gruppo di contenitori altamente disponibile. L'opzione --anti forza tutte le istanze del contenitore nel tuo gruppo ad essere posizionate in un nodo di calcolo fisico separato, che riduce la probabilità che tutti i contenitori in un gruppo abbiano un arresto anomalo a causa di un malfunzionamento dell'hardware. Puoi non essere in grado utilizzare questa opzione con un gruppo di grandi dimensioni perché ogni regione e organizzazione Bluemix dispone di una serie limitata di nodi di calcolo disponibili per la distribuzione. Se la tua distribuzione non avviene, riduci il numero di istanze del contenitore nel gruppo o rimuovi l'opzione --anti.
--auto
(Facoltativo) Quando il ripristino automatico è abilitato per un gruppo di contenitori e il gruppo è stato in esecuzione per 10 minuti, lo stato del programma di bilanciamento del carico inizia a controllare con regolarità l'integrità di ogni istanza del contenitore nel gruppo tramite le richieste HTTP. Se un'istanza del contenitore non risponde entro 100 secondi, viene contrassegnata come inattiva. Le istanze del contenitore inattive vengono rimosse dal gruppo e ricreate dal ripristino automatico. Il ripristino automatico tenta di ripristinare le istanze del contenitore in un gruppo per 3 volte. Dopo il terzo tentativo, il ripristino automatico non ripristina alcuna istanza del contenitore nel gruppo per 60 minuti. Dopo 60 minuti, il processo di ripristino automatico si riavvia nuovamente.Per ulteriori informazioni sul controllo di integrità, consulta l'opzione --http_monitor_enabled.
CMD
(Facoltativo) Il comando e gli argomenti vengono trasmessi al gruppo di contenitori da eseguire. Questo comando deve essere un comando a esecuzione prolungata. Non utilizzare un comando di breve durata, ad esempio /bin/date, in quanto potrebbe provocare un arresto anomalo del contenitore.
Esempio di comandi a lunga esecuzione che utilizzano l'immagine ibmnode fornita da IBM Bluemix Container Service:
cf ic group create --name mio_gruppo_di_contenitori registry.DomainName/ibmnode ping localhost
cf ic group create --name mio_gruppo_di_contenitori registry.DomainName/ibmnode -- tail -f /dev/null
Nota: Il CMD e i suoi argomenti devono trovarsi alla fine della riga di comando cf ic group create.
-d DOMAIN, --domain DOMAIN
(Facoltativo) Il dominio di sistema predefinito è AppDomainName e fornisce già un certificato SSL; puoi quindi accedere ai tuoi gruppi di contenitori con HTTPS senza alcuna configurazione aggiuntiva. Per utilizzare un dominio personalizzato, devi registrarlo su un server DNS pubblico, configurarlo in Bluemix e associarlo quindi al dominio di sistema Bluemix sul server DNS pubblico. Dopo aver associato il tuo dominio personalizzato al dominio di sistema Bluemix, le richieste per il tuo dominio personalizzato vengono instradate alla tua applicazione in Bluemix. Quando crei un dominio personalizzato, non includere caratteri di sottolineatura (_) nel nome del dominio. Per ulteriori informazioni sui domini personalizzati, vedi Creazione e utilizzo di un dominio personalizzato. Per rendere sicuro il tuo dominio personalizzato, carica un certificato SSL in modo che sia possibile accedere ai gruppi di contenitori con HTTPS.

L'host e il dominio, combinati, formano l'URL di rotta pubblico completo, come http://mycontainerhost.AppDomainName, e devono essere univoci in Bluemix. Quando esamini i dettagli di un gruppo di contenitori con il comando cf ic group inspect, l'host e il dominio vengono elencati insieme come rotta.

--desired DESIRED
(Facoltativo) Il numero di istanze nel tuo gruppo. Il valore predefinito è 2. Il numero di istanze del gruppo di contenitori che richiede la tua applicazione dipende dal carico di lavoro previsto e dal tempo nel quale una richiesta può essere elaborata dall'applicazione. Ad esempio, il multitasking o la connettività di rete influisce su quante istanze possono essere richieste per la tua applicazione. Per determinare il numero di istanze desiderato, distribuisci un singolo contenitore in Bluemix che esegue la tua applicazione ed esegue un un test di carico su questo contenitore. Se l'applicazione può gestire, ad esempio, 100 richiesta al secondo, ma prevedi un carico di lavoro di 300 richieste al secondo, il numero desiderato di istanze per il gruppo di contenitori è 3. Per rendere il tuo gruppo di contenitori altamente disponibile e con maggiore resilenza a un malfunzionamento, considera di includere più istanze rispetto al numero minimo per gestire il carico di lavoro previsto. Più istanze possono gestire il carico di lavoro nel caso di un altro arresto anomalo di un'istanza. Per garantire il ripristino automatico di un'istanza con un malfunzionamento senza influire sul carico di lavoro, includi un'ulteriore istanza. Per la protezione da due malfunzionamenti simultanei, includi due ulteriori istanze. Questa configurazione è un modello N+2 dove N è il numero di istanze per gestire le richieste e +2 sono le due istanze aggiuntive.
Importante: Un gruppo di contenitori deve includere almeno 2 istanze del contenitore. Se includi solo 1 istanza nel gruppo e tale istanza diventa non disponibile, i tuoi utenti possono riscontrare dei tempi di inattività per l'applicazione.

-n HOST, --hostname HOST
(Facoltativo) Il nome host, ad esempio mycontainerhost. Non includere i caratteri di sottolineatura (_) nel nome host. L'host e il dominio, combinati, formano l'URL di rotta pubblico completo, come ad esempio http://mycontainerhost.AppDomainName. Quando esamini i dettagli di un gruppo di contenitori con il comando cf ic group inspect, l'host e il dominio vengono elencati insieme come rotta.
--name NAME
(Obbligatorio) Assegnare un nome al gruppo. -n è stato dichiarato obsoleto.
Suggerimento: Il nome contenitore deve iniziare con una lettera e puoi quindi includere lettere maiuscole e minuscole, numeri, punti (.), caratteri di sottolineatura (_) o trattini (-).
-p PORT, --publish PORT

(Obbligatorio) Esporre la porta per il traffico di rete privata e HTTP. Se non esponi una porta, il gruppo di contenitori viene eseguito, ma è impossibile accedervi tramite il programma di bilanciamento del carico di lavoro nella rete privata o pubblica.

Per i gruppi di contenitori, non è possibile includere più porte. Se sono elencate più porte, viene esposta solo l'ultima porta. Le altre porte vengono ignorate dal comando.

Quando esegui il bind a un servizio, i contenitori nel tuo gruppo devono essere in ascolto per il traffico HTTP nella porta presentata del gruppo. Le porte non HTTP non possono essere esposte pubblicamente. Quando il traffico HTTPS arriva alla porta esposta, il (Go)Router completa la terminazione HTTPS. Quindi, viene utilizzato il protocollo HTTP sulla rete virtuale privata tra il (Go)Router e i contenitori. Quando esegui il bind a un indirizzo IP, i contenitori nel tuo gruppo possono essere in ascolto per il traffico HTTP o HTTPs ma possono ancora esporre solo una porta.
Se è specificata una porta nel Dockerfile per l'immagine che stai utilizzando, includi tale porta.
Suggerimento:
  • Per l'immagine Liberty Server certificata da IBM o una versione modificata di questa immagine, immetti la porta 9080.
  • Per l'immagine Node.js certificata da IBM o una versione modificata di questa immagine, immetti una porta qualsiasi, quale ad esempio la porta 8000.
--session_affinity

(Facoltativo) Il valore predefinito è false. Quando --http_monitor_enabled è impostata su true, puoi impostare l'opzione --session_affinity su true per consentire a IBM Bluemix Container Service di creare e gestire un cookie della sessione al posto dell'applicazione. Questa opzione non è disponibile se --http_monitor_enabled è impostata su false o se stai utilizzando TCP per la comunicazione invece di HTTP.

--volume VOLUME:/DIRECTORY_PATH[:ro]
(Facoltativo) Collega un volume a un contenitore specificando i dettagli nel seguente formato VOLUME:/DIRECTORY_PATH[:ro].
  • VOLUME: l'ID o nome del volume.
  • DIRECTORY_PATH: il percorso assoluto alla directory nel contenitore.
  • ro: facoltativo. Se si specifica ro, il volume sarà di sola lettura anziché di lettura/scrittura come da impostazione predefinita.

Dopo che una dimensione del contenitore è stata assegnata, non è possibile modificare il valore. Le dimensioni disponibili in IBM Bluemix Container Service sono:
  • Pico (64 MB di memoria, 4 GB di spazio su disco)
  • Nano (128 MB di memoria, 8 GB di spazio su disco)
  • Micro (256 MB di memoria, 16 GB di spazio su disco)
  • Tiny (512 MB di memoria, 32 GB di spazio su disco)
  • Small (1024 MB di memoria, 64 GB di spazio su disco)
  • Medium (2048 MB di memoria, 128 GB di spazio su disco)
  • Large (4096 MB di memoria, 256 GB di spazio su disco)
  • X-Large (8192 MB di memoria, 512 GB di spazio su disco)
  • 2X-Large (16384 MB di memoria, 1 TB di spazio su disco)

(Facoltativo) Se vuoi rendere la tua applicazione accessibile da Internet, devi esporre una porta pubblica. Tutte le altre porte pubbliche sono chiuse e non possono essere utilizzate per accedere alla tua applicazione da internet. Puoi includere più porte con più opzioni -p. Le porte non possono essere associate o inoltrate. Se non esponi una porta, il tuo contenitore è accessibile solo dalla rete del contenitore privata. Puoi utilizzare l'indirizzo IP assegnato per comunicare con il tuo contenitore nella rete privata.

  • Per l'immagine Node.js certificata da IBM o una versione modificata di questa immagine, immetti una porta qualsiasi, quale ad esempio la porta 8000.

  • -p 9080/udp -o 8000/tcp

I gruppi di contenitori sono associati a una rotta pubblica che già include un certificato SSL, per cui puoi accedere al tuo gruppo di contenitori con HTTPS senza alcuna ulteriore configurazione. Per ulteriori informazioni, consulta Creazione di un gruppo di contenitori con la GUI Bluemix.
